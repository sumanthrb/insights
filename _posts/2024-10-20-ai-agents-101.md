I've had the same conversation at least a dozen times in the past few months. Someone mentions AI agents, the room nods enthusiastically, and then five minutes later it becomes clear that nobody, including the person who brought it up, can precisely explain what an agent actually *is* or how it works mechanically.

So let me try to fix that. No hype, no hand-waving. Just the actual mechanics.

## First, What's Wrong With a Regular LLM?

A standard LLM interaction is simple: we send a prompt, we get a response. One round trip. The model doesn't remember the previous conversation unless we explicitly pass it in. It can't browse the web, run code, or check our database. It just generates text.

That's genuinely useful,  but it hits a ceiling fast. Real-world tasks rarely fit into a single prompt & response cycle. Researching a topic, debugging a production issue, writing and testing code; these involve multiple steps, external information, and decisions that depend on intermediate results.

That's the gap agents are designed to fill.

## What Makes Something an Agent

An AI agent is a system that can *loop*. Specifically, it:

1. **Perceives** its current situation. the user's goal, previous steps, tool results
2. **Reasons** about what to do next
3. **Acts**, usually by calling a tool or producing output
4. **Observes** what happened
5. **Repeats** until the job is done

The loop is the thing. A chatbot responds. An agent *iterates*.

```
Perceive → Reason → Act → Observe → Perceive (loop)

```

It sounds simple, and honestly the core idea *is* simple. The complexity comes from doing it reliably at scale — but we'll get to that.

## The Four Things Every Agent Needs

Strip away the framework marketing and every agent is built from the same four parts.

### 1. The LLM (The Brain)

The language model does the reasoning. It looks at the current state: the goal, conversation history, available tools, what's been tried and decides what to do next.

One thing worth internalizing early: **the LLM itself has no memory**. Every time we call it, we're starting fresh. All context has to be passed in explicitly with each request. The agent infrastructure around it is largely responsible for managing that state.

```python
response = llm.complete(
    system_prompt=agent_instructions,
    messages=conversation_history,
    tools=available_tools
)
```

### 2. Memory

Since the model forgets everything between calls, we need external memory systems. There are a few flavors:

| Memory Type | What It Stores | Example |
|---|---|---|
| **In-context** | The active conversation | Recent messages passed directly in the prompt |
| **Episodic** | Past interactions | "Last time we asked about X..." |
| **Semantic** | Facts and knowledge | A vector database with our company's docs |
| **Procedural** | How to behave | System prompts, rules, instructions |

Most production agents combine in-context memory (what fits in the current window) with semantic memory (a vector DB for retrieving relevant information on demand). The others come in as the use case gets more sophisticated.

### 3. Tools

This is what separates an agent from a chatbot. Tools let the agent actually *do things* in the world : search the web, run code, query a database, call an API, send a message.

We define tools as structured descriptions, and when the LLM decides it needs one, it generates a tool call. Our code runs it and feeds the result back in.

```python
tools = [
    {
        "name": "search_web",
        "description": "Search the internet for current information",
        "parameters": {
            "query": {"type": "string", "description": "Search query"}
        }
    },
    {
        "name": "execute_code",
        "description": "Run Python code and return the output",
        "parameters": {
            "code": {"type": "string", "description": "Python code to run"}
        }
    }
]
```

The quality of our tool descriptions matters a lot here. A vague description leads to the model using tools incorrectly or not at all. Write them like we're explaining the tool to a smart engineer who's never seen our codebase.

### 4. The Orchestrator

The orchestrator is the loop itself, so the code that ties everything together. It manages state, routes tool calls to the right functions, handles errors, and decides when the agent is done.

Here's what a minimal orchestrator looks like:

```python
def run_agent(user_goal: str) -> str:
    messages = [{"role": "user", "content": user_goal}]

    while True:
        response = llm.complete(messages=messages, tools=tools)

        if response.tool_calls:
            for tool_call in response.tool_calls:
                result = execute_tool(tool_call.name, tool_call.arguments)
                messages.append({"role": "tool", "content": result})
        else:
            return response.content
```

That's it. Every major agentic framework (like LangGraph, CrewAI, AutoGen) is a more sophisticated version of this loop with better error handling, observability, and state management built in.

## The ReAct Pattern

Most agents we'll encounter in the wild use a pattern called **ReAct** (Reasoning + Acting), from a 2022 Google Research paper. The core idea is to make the model's reasoning explicit before it takes any action.

Instead of jumping straight to tool calls, the model thinks out loud first:

```
Thought: I need to find current EC2 pricing for a t3.medium.
Action: search_web("AWS EC2 t3.medium on-demand pricing 2026")
Observation: $0.0416/hour in us-east-1.

Thought: Now I can calculate the monthly cost.
Action: calculate(0.0416 * 24 * 30)
Observation: 29.952

Thought: I have enough to answer now.
Final Answer: A t3.medium running 24/7 costs roughly $29.95/month.
```

Why does this work better? Because forcing explicit reasoning before action reduces impulsive tool calls and makes the agent's decision-making transparent. When something goes wrong and it will !! we can see exactly where the reasoning broke down rather than staring at a wrong final answer with no idea how we got there.

## When One Agent Isn't Enough

For simple tasks, a single agent works fine. But complex tasks, the kind actually worth automating, often benefit from splitting responsibilities across multiple specialized agents:

![AI Agent Loop Diagram]({{ '/images/orky.png' | relative_url }})

A research agent handles web search and synthesis. A coding agent writes and tests code. A review agent checks for correctness, security issues, or policy violations. An orchestrator coordinates them and assembles the final output.

This mirrors how good engineering teams work, specialists collaborating, not one person trying to do everything at once.

## Where Agents Actually Break

I'd be doing we a disservice if I only covered the happy path. Agents fail in specific, predictable ways, and knowing them upfront saves a lot of painful debugging later.

**Hallucinated tool calls** : The model will sometimes call tools with incorrect arguments, or invent tools that don't exist. Always validate inputs before execution and handle errors gracefully.

**Infinite loops** : Without a stopping condition, an agent chasing a vague goal can spin forever. Set a hard step limit:

```python
MAX_STEPS = 20
step = 0

while step < MAX_STEPS:
    response = llm.complete(messages=messages, tools=tools)
    step += 1
    if not response.tool_calls:
        break
```

**Context window overflow** : Long-running agents accumulate a lot of context. After enough steps, we'll hit the model's token limit. Use summarization, context pruning, or external memory to keep the window manageable.

**Compounding errors** : Each step builds on the last. A wrong assumption in step 2 can quietly snowball into a completely broken result by step 10. Add validation checkpoints between critical steps, especially before any irreversible actions.

## A Working Example

Here's a minimal but complete agent using the Anthropic API;  no frameworks, no abstractions, just the raw mechanics:

```python
import anthropic

client = anthropic.Anthropic()

tools = [
    {
        "name": "get_weather",
        "description": "Get current weather for a given city",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "City name"}
            },
            "required": ["city"]
        }
    }
]

def get_weather(city: str) -> str:
    # Replace with a real weather API call
    return f"Sunny, 24°C in {city}"

messages = [{"role": "user", "content": "What's the weather like in Hyderabad?"}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "tool_use":
        tool_use = next(b for b in response.content if b.type == "tool_use")
        result = get_weather(**tool_use.input)

        messages.append({"role": "assistant", "content": response.content})
        messages.append({
            "role": "user",
            "content": [{
                "type": "tool_result",
                "tool_use_id": tool_use.id,
                "content": result
            }]
        })
    else:
        print(response.content[0].text)
        break
```

Run this and we'll see the loop in action. The model calls the tool, gets the result, uses it to form a final answer. Add more tools, a richer system prompt, and better memory management and we have the foundation of a real production agent.

## Which Framework Should We Use?

| Framework | Good For | Overhead |
|---|---|---|
| **Raw API** | Learning, full control, simple agents | Minimal |
| **LangChain** | Quick prototypes, huge plugin ecosystem | Medium |
| **LangGraph** | Complex stateful workflows, branching logic | Higher |
| **CrewAI** | Multi-agent role-based collaboration | Medium |
| **AutoGen** | Conversational multi-agent research | Medium |

<br>
My honest take: **start with the raw API**. Build a few agents from scratch before reaching for a framework. we'll understand what frameworks are actually doing for us and more importantly, what they're hiding from us. When we eventually hit the limits of raw API work, we'll reach for a framework knowing exactly why.

## The One Thing Worth Remembering

All the complexity:  memory systems, orchestration patterns, multi-agent coordination, ReAct loops is scaffolding around one simple idea:

> **An agent is a while loop around an LLM with tools.**

Once that clicks, we can reason clearly about any agent system. We can predict where it'll struggle, trace why it behaved unexpectedly, and know what to fix when it breaks.
